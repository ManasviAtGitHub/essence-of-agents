# 2026-07-11

## resolve got slow under load

- **Ask:** redirects were fine in isolation but p95 latency climbed under load; it tracked
  the disk, not the lookup.
- **Summary:** `resolve` was doing a synchronous `store.put` to increment `hits` *before*
  returning the URL -- a disk write on the hottest read path in the system.
- **Decision:** return the URL first, then `bump(code)` separately (in production: queued and
  flushed off the hot path). A dropped increment is acceptable; a slow redirect is not, so
  `hits` is now explicitly eventually-consistent.
- **Why:** resolve is the one thing that has to stay fast. Recorded as `lessons.md` #3, the
  "resolve stays cheap" non-negotiable (#4), and the eventually-consistent note in
  `decisions.md`.
